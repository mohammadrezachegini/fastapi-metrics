# FastAPI Metrics Microservice

A minimal FastAPI microservice deployed on Kubernetes, built to demonstrate horizontal pod autoscaling (HPA) in action. Exposes a health check endpoint and a Prometheus-compatible metrics endpoint, scales automatically from 1 to 5 replicas under CPU pressure.

Built as part of the [Root Access Newsletter](https://www.linkedin.com/in/mohammadrezachegini/) home lab series.

Live at `fastapi.rezaops.com`.

---

## Stack

| Layer | Technology |
|---|---|
| Framework | FastAPI + uvicorn |
| Metrics | prometheus-fastapi-instrumentator |
| Container | Docker, registry.rezaops.com |
| Orchestration | Kubernetes (K3s) |
| Autoscaling | HorizontalPodAutoscaler (HPA) |
| Metrics source | metrics-server (K3s built-in) |
| Ingress | NGINX Ingress Controller |

## Endpoints

```
GET /          service info and available endpoints
GET /health    health check with pod hostname
GET /metrics   Prometheus-format metrics (auto-generated)
GET /docs      Swagger UI (FastAPI built-in)
```

### Health response

```json
{
  "status": "healthy",
  "service": "fastapi-metrics",
  "version": "v1",
  "hostname": "fastapi-metrics-7748cd54c4-zpmj7"
}
```

The `hostname` field returns the pod's hostname — useful for confirming that NGINX Ingress is actually distributing traffic across multiple replicas when scaled up.

## Kubernetes Deployment

Requires a running K3s cluster with metrics-server available (K3s includes it by default).

```bash
# Apply all manifests (Deployment, Service, Ingress, HPA)
kubectl apply -f k8s-manifest.yaml

# Check pod
kubectl get pods -n fastapi-metrics

# Check HPA status
kubectl get hpa -n fastapi-metrics
```

Expected HPA output at idle:

```
NAME                  REFERENCE                    TARGETS       MINPODS   MAXPODS   REPLICAS
fastapi-metrics-hpa   Deployment/fastapi-metrics   cpu: 4%/50%   1         5         1
```

## HPA Configuration

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: fastapi-metrics
  minReplicas: 1
  maxReplicas: 5
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
```

**Important:** `resources.requests.cpu` must be defined on the container for HPA to function. Without it, HPA has no baseline to calculate utilization percentage against and will show `<unknown>` for targets.

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "64Mi"
```

## Load Testing and Autoscaling Demo

```bash
# Install Apache Bench
sudo apt install -y apache2-utils

# Watch HPA in one terminal
kubectl get hpa -n fastapi-metrics -w

# Watch pods in another terminal
kubectl get pods -n fastapi-metrics -w

# Run load test (use IP directly to bypass DNS if needed)
ab -n 100000 -c 50 -H "Host: fastapi.rezaops.com" http://<INGRESS_IP>/health
```

### Observed results (live from the cluster)

```
cpu: 4%/50%     replicas: 1    idle baseline
cpu: 50%/50%    replicas: 1    threshold reached
cpu: 201%/50%   replicas: 1    scale-up triggered
cpu: 201%/50%   replicas: 5    scaled to max
cpu: 5%/50%     replicas: 5    load test ended
(stabilization window ~5 min)
cpu: 5%/50%     replicas: 1    scaled back down
```

### Verify load distribution across replicas

```bash
for i in {1..10}; do
  curl -s -H "Host: fastapi.rezaops.com" http://<INGRESS_IP>/health | grep hostname
done
```

Expected: multiple distinct pod hostnames across the 10 requests, confirming NGINX is round-robin load-balancing across all live replicas.

## Prometheus Integration

Pod annotations tell Prometheus to auto-discover and scrape this service:

```yaml
annotations:
  prometheus.io/scrape: "true"
  prometheus.io/port: "8000"
  prometheus.io/path: "/metrics"
```

The `/metrics` endpoint is auto-generated by `prometheus-fastapi-instrumentator` — tracks request counts, latencies, and status codes for every route with no manual metric code.

## Key Design Decision

`--workers 1` in the uvicorn command is deliberate. A single worker per pod keeps CPU load per replica predictable, making the relationship between incoming traffic and pod count the dominant observable effect. This makes HPA scaling behavior clean and easy to demonstrate — if the app absorbed load internally via multiple workers, the CPU pressure on any single pod would be lower and scaling would be harder to trigger and observe.
